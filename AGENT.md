这个文件是给 AI 开发代理看的，不是项目介绍。AI 做任何开发、修改、编译、测试，都必须先看这个文件，再看对应需求文档和实际代码。
目标很简单：只看本文档、需求文档和实际代码，就能把功能做对、编译通过、测试清楚，不能靠口头猜。
默认用中文交流。代码名、路径、命令、target 名称、类名函数名保留原文。
禁止说“肯定是”“我觉得”“应该是”来代替验证。如果想这么说，就直接看代码、查日志、跑编译、跑测试，用结果说话。
禁止操作 git commit。可以用 git status、git diff、git log、git show 查信息，不可以用 git 修改文件，不可以用 git 丢弃修改。禁止 git reset --hard、git checkout -- 这类会覆盖工作区的命令。
提交信息必须使用中文。专题开发提交信息优先采用：
专题<编号>按文档落地<专题目标>
提交正文用 3-5 条说明：
- 按哪个 docs/<专题目录> 文档落地
- 改了哪些核心模块
- 补了哪些测试 target
- 用了哪些 bat 验证
- 如有未验证风险，简要说明
禁止只写英文 conventional commit，除非用户明确要求。
不要在 py 里面套 py。已有脚本能直接运行就直接运行，不要为了执行一个脚本再临时写一个 Python 包一层。如果确实需要新增脚本，必须放在项目约定目录，并说明用途，不要随手丢在根目录。
开发必须文档驱动。每次新功能、修复、接口调整、数据流程调整，先看文档，再改文档，再改代码。文档按过去项目格式写，不另起一套风格。
文档顺序固定：
01-PRD.md 写需求，必须最先写或更新。必须写清楚功能目标、输入、输出、边界、异常情况、验收标准。PRD 要让 AI 知道要做什么、做到什么程度、哪些东西不能顺手改。PRD 未经用户确认前，禁止写或更新 02-ARCH.md、03-TEST-CASES.md。
02-ARCH.md 写架构。必须在 PRD 经用户确认后才能写或更新。写 ARCH 时必须深入阅读实际代码，确认模块职责、数据从哪里来、流到哪里去、接口位置、调用链、现有约束、谁负责处理、接口怎么连。ARCH 只能覆盖 PRD 已确认的目标、输入、输出、边界、异常情况和验收标准，不能引入 PRD 没有确认的新功能、新数据源、新交互、新接口或新范围。ARCH 要让 AI 知道代码应该放哪里，不能把逻辑塞错层。
03-TEST-CASES.md 写测试。必须在 PRD 经用户确认后才能写或更新。写 TEST-CASES 时必须深入阅读实际代码，确认已有测试框架、测试入口、已有用例、验证脚本和可测边界。TEST-CASES 只能覆盖 PRD 已确认的目标、输入、输出、边界、异常情况和验收标准，不能引入 PRD 没有确认的新功能、新数据源、新交互、新接口或新测试范围。测试文档要能直接指导测试代码和验证流程。
PRD 确认后，02-ARCH.md 和 03-TEST-CASES.md 可以并行推进，但都只能对照已确认的 PRD 和实际代码写。写 ARCH 或 TEST-CASES 时，如果发现 PRD 缺失、矛盾或范围不清，必须先回到 PRD 修改并再次交给用户确认，不能直接在 ARCH 或 TEST-CASES 里补需求。禁止先写 ARCH 或 TEST-CASES 再倒推 PRD。
标准文档流程：
第一步：阅读现有需求文档和相关代码，只更新 PRD。
第二步：等待用户确认 PRD。
第三步：PRD 确认后，再基于 PRD 和实际代码更新 ARCH、TEST-CASES。
第四步：三份文档一致后，才允许进入代码修改。
文档必须能驱动开发。不要写空话。AI 只看 01、02、03 和实际代码时，必须能判断：
要改什么；
不改什么；
数据源是哪一个；
哪个模块负责；
怎么编译；
怎么测试；
怎么算完成。
必须保证数据源唯一。同一份核心数据只能有一个权威来源。禁止多个模块各自读、各自解析、各自维护一份同样的数据。需要数据就通过统一接口取，不要绕开主流程。
职责必须独立。UI 只做展示和交互，不直接承担业务计算、数据解析、硬件通信、文件存储这类职责。业务逻辑、数据访问、算法计算、硬件通信要分清楚。不要把跨模块流程硬编码在界面层。
方案软硬程度要合适。不要为了简单把所有逻辑堆进一个类，也不要为了“架构好看”搞一堆没有实际价值的抽象。优先跟项目现有结构、命名、风格、工具链保持一致。
每次需要编译，都必须通过 bat 脚本编译。用户说“编译”时，默认含义是使用项目 scripts 或测试目录里的 bat 脚本驱动 CMakeLists，不要直接手敲 cmake --build。不要把临时手敲的 cmake 命令当成最终交付方式。需要新增或调整构建流程，就更新对应 bat 脚本。
CMakeLists.txt 的 target 要写完整。客户端程序、依赖库、资源文件、输出路径、测试 target 都要配置清楚。不要靠 IDE 临时配置补齐构建。
发行版客户端必须是最新编译结果。改完代码后，如果任务涉及客户端运行或交付，必须重新跑发行版构建脚本，不能拿旧 exe 当结果。
单元测试必须单独配置 CMake 编译。测试 target 和客户端发行 target 分开，不要混在一起。新增测试时，同步维护测试 CMake 配置和测试编译 bat。
专题开发必须优先运行本专题相关的最小测试 target。scripts\test\test_unit.bat 只允许构建并运行当前单元测试入口或本专题新增 target，不得默认运行全部已挂载 CTest，不得默认跑 RUN_TESTS。全量回归必须使用独立入口 scripts\test\test_all_enabled.bat，且只有用户明确要求全量回归，或本次修改影响公共基础设施时才运行。遇到全量历史失败项时，必须区分新增失败和既有失败，不能把既有失败当成本专题阻塞。
修改代码后的正确流程：
先更新或确认 01-PRD.md。
再更新或确认 02-ARCH.md。
再更新或确认 03-TEST-CASES.md。
然后改代码。
然后检查 CMakeLists.txt target。
然后用 bat 编译客户端。
然后用单独 bat 编译和运行单元测试。
最后说明改了什么、怎么编译、怎么测试、还有什么没验证。
当遇到前端、后端、服务端口、客户端端口被占用时，必须先查占用端口的进程 PID，确认这个进程属于当前项目后，才允许关闭。禁止不查来源就杀进程。禁止一遇到端口占用就换端口。固定端口被占用时，要处理占用进程，不要绕过去。
日志要按项目已有路径查。先看主日志，再根据 session、worker、module 日志继续查。不要只看一行报错就下结论。
可以使用项目内 tmp 或约定临时目录。禁止把临时文件放到系统根目录、桌面、下载目录这类不受项目管理的位置。临时文件用完要清理，除非明确说明它是调试产物。
